# 구현 스킬

## 트리거

FE 스펙이 `승인됨` 상태가 된 후, 민서가 구현을 요청했을 때 사용한다.

## 전제 조건

- 대상 피처의 FE 스펙(`docs/spec/<공통 또는 페이지>/XX-피처명.md`)이 존재해야 한다
- 스펙 상태가 **승인됨**이어야 한다
- 승인되지 않은 스펙으로는 구현을 시작하지 않고, 민서에게 승인 여부를 확인한다

## 구현 절차

1. 해당 FE 스펙 문서를 읽고 구현 범위를 파악한다
2. 스펙의 **파일 구조** 섹션에 명시된 파일을 기준으로 작업한다
3. 스펙의 **컴포넌트 명세**에 정의된 props, 동작을 그대로 구현한다
4. 스펙의 **API 연동** 섹션에 정의된 인터페이스와 에러 처리를 따른다
5. 스타일은 **https://tossinvest.com** 실제 서비스를 참고하여 클론한다
6. 스펙에 없는 케이스를 코드로 보완(공백 보완)했다면 아래 **기록 = 반영** 절차를 수행한다
7. 구현 완료 후 **구현 보고서**를 출력한다
8. 보고서 출력 후 민서에게 리뷰 진행 여부를 확인한다

## 공백 보완 시: 기록 = 반영 (ADR-002)

스펙에 없는 케이스를 코드로 보완(= 임의 판단)할 때는 메모로만 남기지 않고 **같은 브랜치에서 대상 문서를 함께 고친다.** 이 문서 변경이 곧 게이트(ADR-002 item 2)의 "docs 변경됨" 증거가 된다.

1. 보완 내용을 반영할 대상 문서를 고른다
   - 비즈니스 규칙·시나리오·예외 처리 공백 → `docs/planning/...`
   - 컴포넌트 명세·API·상태 관리 공백 → `docs/spec/...`
2. 해당 문서 **본문**을 보완 내용에 맞게 직접 고친다 (코드와 문서가 일치하도록)
3. 그 문서의 **변경 이력** 표에 `[제안]` 행을 추가한다 (날짜·변경 내용·현재 브랜치). 표가 없으면 통합 형식으로 신설한다
4. 이 문서 변경을 구현 코드와 **같은 브랜치/PR**에 포함한다
5. 구현 보고서의 "임의 판단한 것"에 어떤 문서에 어떤 `[제안]`을 넣었는지 명시한다

> 머지 시 `[제안] → [확정]` 전환은 게이트 워크플로(item 2)가 처리한다. 게이트 도입 전까지는 민서가 PR 승인 시 수동 전환한다. implement는 `[제안]`까지만 만든다.

## src를 바꿨지만 문서 영향이 없을 때 (no-spec-change)

props·동작·API·비즈니스 규칙 등 **관찰 가능한 동작이 바뀌지 않는** src 변경(순수 리팩터링, 스타일·타입 정리, 스펙대로 고친 버그, 성능 최적화 등)은 기획서/스펙을 고칠 필요가 없다. 이 경우:

1. 문서를 억지로 고치지 않는다 (게이트 통과용 형식적 편집 금지)
2. 구현 보고서에 **"no-spec-change 라벨 대상"** 임을 명시하고, 왜 문서 영향이 없는지(동작 보존 근거)를 적는다
3. 게이트(2a)는 `no-spec-change` 라벨이 있어야 src-only 변경을 통과시킨다. **라벨의 실제 부착·정당성 판단은 민서(리뷰어)** 가 한다

> 경계: 리팩터·버그 수정이라도 **관찰 가능한 동작이 바뀌면** 라벨 대상이 아니라 위 "기록 = 반영" 대상이다.

## 구현 보고서 형식

```markdown
## 구현 보고서

### 구현한 것
- ...

### 임의 판단한 것
- ...

### 못한 것
- ...
```

> "임의 판단한 것"의 각 항목은 위 **기록 = 반영** 절차로 만든 문서 `[제안]`과 1:1로 대응해야 한다. 대응되는 `[제안]`이 없는 임의 판단은 미반영이므로 절차를 다시 수행한다.
> src를 바꿨지만 동작이 보존돼 문서 영향이 없으면, 보고서에 `no-spec-change 라벨 대상 — (동작 보존 근거)`를 함께 명시한다.

## 규칙

- AGENTS.md와 .codex/codex.md의 핵심 원칙을 따른다
- 스타일(색상, 간격, 폰트, 레이아웃)은 실제 토스증권 사이트를 기준으로 클론한다
