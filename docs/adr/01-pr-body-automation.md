# ADR-001: PR 본문 자동 생성·갱신 (GitHub Actions)

> 작성일: 2026-05-28
> 상태: 제안됨 | **채택됨** | 폐기됨
> 관련 이슈: #14 (GitHub Actions를 활용한 자동화 구축 1 ‐ PR 생성)

## 맥락

- 현재 PR 본문은 로컬에서 Claude가 커밋을 분석해 수기로 작성한 뒤 생성한다. 세션·사람에 의존적이라 형식 불일치나 누락이 생길 수 있다.
- PR을 올린 뒤 push로 커밋이 추가돼도 본문이 자동 갱신되지 않아, Commits 표가 실제 커밋과 어긋날 수 있다. (pull-request 스킬의 "커밋 수 일치" 규칙을 사람이 매번 손으로 지켜야 한다.)
- 또한 이 워크플로의 **브랜치/base 감지·커밋 파싱·PR 본문 플러밍**은 향후 기획-개발 싱크 자동 게이트(이슈 #12)가 그대로 올라탈 토대가 된다.

## 고려한 선택지

### 선택지 1: 로컬 스킬 유지 (현행)
- 장점: 추가 인프라가 필요 없다
- 단점: 세션·사람에 의존한다. push 후 본문이 갱신되지 않는다. 누락 위험이 남는다

### 선택지 2: GitHub Actions로 PR 본문 자동 생성·갱신
- 장점: PR open/push마다 자동 동작, 사람 개입 최소화, 싱크 게이트(#12)의 토대가 된다
- 단점: Actions 인프라와 토큰 권한 관리가 필요하다

### 선택지 3: 클라이언트 git hook (pre-push 등)
- 장점: 로컬에서 즉시 동작한다
- 단점: 사람마다 설치해야 해서 규율에 의존한다. 원격 PR 본문에는 접근이 제한된다

## 결정

**선택지 2를 채택한다.** GitHub Actions 워크플로로 PR open 및 push 시 본문의 Commits 영역을 자동 생성·갱신한다.

- **트리거**: `pull_request` 이벤트의 `opened`, `synchronize`, `reopened`
- **base/head 확인**: PR 이벤트 페이로드의 `github.event.pull_request.base.ref` / `head.ref`로 확인한다
- **본문 생성 규칙**: `base..head` 커밋을 **오래된 순**으로 읽어 Commits 표를 만든다 (pull-request 스킬의 본문 형식·정렬 규칙을 따른다)
- **자동/수동 영역 분리**: 본문에 마커를 두고 **마커 내부만 갱신**한다. Summary 등 작성자가 직접 입력한 텍스트는 보존한다.
  ```
  <!-- AUTO:commits:start -->
  (이 영역만 워크플로가 갱신)
  <!-- AUTO:commits:end -->
  ```
  마커가 없으면 본문 끝에 자동 영역을 1회 추가하고, 그 외 기존 본문은 건드리지 않는다.
- **실패 처리**: 워크플로가 실패하면 PR에 다음 코멘트를 남긴다.
  > `prTemplate action이 실패했습니다. PR 본문을 수동으로 입력해주세요.`
- **치명도**: 이 워크플로는 **비차단(advisory)**이다. 실패해도 머지를 막지 않는다. (머지를 막는 것은 이슈 #12의 싱크 게이트이며, 둘은 별도 워크플로·별도 실패 의미로 둔다.)

## 결과

- PR 본문의 Commits 표가 항상 최신 커밋과 일치하여, pull-request 스킬의 수기 검증 부담이 줄어든다.
- 작성자가 직접 쓴 Summary 등은 갱신 과정에서 보존된다.
- 이슈 #12의 자동 게이트가 이 워크플로의 브랜치/커밋 파싱·플러밍을 재사용할 수 있다.
- 비차단 설계라 #12 게이트(차단)와 치명도가 명확히 구분된다.
