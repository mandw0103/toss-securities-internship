# PR 스킬

## 트리거

민서가 PR 생성을 요청했을 때 사용한다.

## 절차

1. `git log`로 현재 브랜치의 커밋 목록을 확인한다
2. 커밋 내용을 분석하여 PR **제목**과 **본문**을 작성한다
3. 민서에게 추천안을 보여주고 **승인을 받는다**
4. 민서가 승인하면 `gh pr create --base main`으로 PR을 생성한다 (**base는 항상 `main`**)

## PR 제목 형식

```
[ISSUE-N] <gitmoji> <한글 설명>
```

- `N`은 현재 브랜치명에서 추출한 이슈 번호이다 (예: `issues/2` → `2`)
- `<gitmoji>`는 커밋 스킬의 분류 기준(📝 / ✨🐛♻️ / 👩‍💻 / 🔧)을 따른다
- 70자 이내로 작성한다

### 여러 카테고리가 섞인 PR의 깃모지 선택 기준

여러 종류의 변경(예: 📝 + 👩‍💻 + 🔧)이 한 PR에 묶일 수 있다. 이 경우 아래 우선순위로 깃모지를 정한다:

1. **PR의 묶음 의도**(왜 이 변경들이 한 PR로 묶였는가)에 가장 부합하는 카테고리를 우선한다
   - 예: 운영 규칙·구조 정비가 묶음 의도이면 🔧
   - 예: 피처 구현 완료가 묶음 의도이면 ✨
2. 동률이거나 모호하면 **변경 파일 수가 가장 많은 카테고리**를 선택한다
3. 그래도 결정이 어려우면 깃모지 후보 2~3개를 제시하고 민서에게 질문한다

## PR 본문 형식

```markdown
## Summary
- 변경 사항 요약 (1~3줄)

## Commits
| 해시 | 메시지 |
|------|--------|
| [`abc1234`](커밋 링크) | 커밋 메시지 |
| [`def5678`](커밋 링크) | 커밋 메시지 |
```

## 규칙

- PR 제목은 위 **PR 제목 형식**을 따른다 (`[ISSUE-N] <gitmoji> <한글 설명>`, 70자 이내)
- 본문에 모든 커밋의 해시, 링크, 메시지를 포함한다
- **PR의 base 브랜치는 무조건 `main`이다.** 다른 브랜치(예: 직전 작업 브랜치, `issues/N`)를 base로 지정하지 않는다
  - `gh pr create` 호출 시 항상 `--base main`을 명시한다
  - 현재 브랜치가 `main`에서 직접 갈라지지 않은 상태(다른 작업 브랜치 위에 쌓인 경우)라도, base는 `main`으로 두고 필요 시 민서에게 리베이스/머지 방법을 확인한다
- **민서의 승인 없이 PR을 생성하지 않는다**
- 추천안을 보여준 뒤 반드시 승인 여부를 확인한다
