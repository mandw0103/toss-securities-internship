# 리뷰 스킬

## 트리거

민서가 구현 결과에 대해 리뷰를 요청했을 때 사용한다.

## 리뷰 범위

현재 사이클 내에서 구현한 내용을 리뷰한다.

## 리뷰 절차

1. 대상 피처의 기획서를 읽는다 (위치: `docs/planning/common/` 또는 `docs/planning/<페이지명>/`)
2. 대상 피처의 FE 스펙을 읽는다 (기획서와 동일한 분류 위치: `docs/spec/common/` 또는 `docs/spec/<페이지명>/`)
3. 구현된 코드를 읽고 스펙과 대조한다
4. 구현 중 **공백 보완**(스펙에 없는 케이스를 코드로 처리)이 있었다면, 그 내용이 **대상 문서 본문 + 변경 이력 `[제안]` 행**에 반영됐는지 확인한다 (ADR-002 "기록=반영"). 미반영이면 문제로 보고한다
5. 구현 보고서에 **no-spec-change 라벨 대상**으로 적힌 src-only 변경은, 관찰 가능한 동작(props·동작·API·비즈니스 규칙)이 정말 안 바뀌었는지 근거를 감사한다 (생략된 2b AI 라벨 감사를 민서가 대신). 타당하면 민서가 PR에 `no-spec-change` 라벨을 부착하고, 타당하지 않으면(실은 동작이 바뀜) 문제로 보고해 기록=반영을 요구한다
6. 실제 토스증권 사이트(https://tossinvest.com)와 스타일을 비교한다
7. 발견된 문제마다 **원인 분류**를 수행한다
8. 아래 **리뷰 보고서** 형식으로 결과를 보고한다

## 원인 분류 기준

발견된 문제는 반드시 아래 4가지 중 하나로 분류한다:

| 분류 | 설명 | 후속 조치 |
|------|------|-----------|
| **기획서 문제** | `docs/planning/` 문서의 내용이 모호하거나 누락된 경우 | 기획서 수정 필요 |
| **스펙 문제** | `docs/spec/` 문서의 기술 명세가 부정확하거나 빠진 경우 | 스펙 수정 필요 |
| **CLAUDE.md 문제** | 운영 규칙이 모순되거나 부족하여 발생한 경우. 공백 보완을 문서에 반영하지 않은 **기록=반영 미준수**도 여기로 분류한다 | CLAUDE.md 또는 스킬 수정 필요. 기록=반영 미준수는 누락된 문서 `[제안]` 반영 |
| **구현 실수** | 스펙대로 구현하지 않은 코드 레벨 버그 | 코드 수정 필요 |

## 리뷰 보고서 형식

```markdown
## 리뷰 보고서

### 대상
- 피처: [피처명]
- 기획서: docs/planning/<위치>/XX-피처명.md
- 스펙: docs/spec/<위치>/XX-피처명.md

### 문제 목록

#### 1. [문제 요약]
- **원인 분류**: 기획서 문제 | 스펙 문제 | CLAUDE.md 문제 | 구현 실수
- **상세**: 무엇이 잘못되었는지 구체적으로 설명
- **근거**: 어떤 문서/코드에서 확인했는지
- **후속 조치 제안**: 어떤 문서 또는 코드를 수정해야 하는지

#### 2. [문제 요약]
- ...

### 문제 없음 확인 항목
- 정상적으로 구현된 항목 나열

### 종합 의견
- 전체적인 리뷰 소견
```

## 규칙

- 리뷰는 **보고까지만** 수행한다. 코드나 문서를 직접 수정하지 않는다
- 모든 문제에 반드시 원인 분류를 붙인다. 분류 없는 문제 보고는 하지 않는다
- 문제가 없는 항목도 명시하여 리뷰 범위를 명확히 한다
- 추측이 아닌 문서와 코드 근거를 기반으로 판단한다
- `no-spec-change` 라벨의 부착·정당성 감사는 **민서(리뷰어)의 책임**이다 (ADR-002 2b 자동 감사를 생략한 대체 장치). 라벨은 민서만 붙인다
